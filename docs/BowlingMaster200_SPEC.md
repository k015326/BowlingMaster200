BowlingMaster200 統合仕様書 v1.2
（正式版・凍結版）
________________________________________
1. プロジェクト概要
   BowlingMaster200 は、
   ・SpareMaster の OCR / スペア解析機能
   ・Trainer180 の ゲーム入力 / 統計分析機能
   を統合した総合ボウリング分析アプリである。
   本プロジェクトでは、
   「OCRコアを壊さずに統合する」
   ことを最優先方針とする。
   OCRは補助機能として位置付け、
   最終的なスコア確定は利用者自身が行う。
________________________________________
2. 開発目的
   従来、
   Trainer180 は手入力専用アプリであった。
   SpareMaster はOCRによるスペア解析専用アプリであった。
   BowlingMaster200 では、
   ・OCR読取
   ・手入力
   ・手修正
   ・ゲーム保存
   ・統計分析
   ・スペア分析
   を単一アプリ内で実現する。
   利用者は、
   ボウリング場でのスコア管理から、
   帰宅後の分析までを、
   BowlingMaster200のみで完結できることを目標とする。
________________________________________
3. 最重要方針（絶対遵守）
   3-1. OCRコアを編集しない
   SpareMaster 側の以下は LOCKED とする。
   対象
   ・util/ocr/
   ・util/layout/
   ・util/symbol/
   ・ScoreSheetLayout
   ・ScoreSheetLayoutAnalyzer
   ・OcrInputBitmapPipeline
   ・SymbolRowAnalyzer
   ・CapturedImageOcrCandidate
   ・SpareOutcomeRules
   原則
   ・修正禁止
   ・ロジック変更禁止
   ・閾値変更禁止
   ・リファクタリング禁止
   必要な調整は、
   ・Adapter層
   ・Mapper層
   ・UI層
   で吸収する。
________________________________________
3-2. LOCKED_STATE.md の継承
SpareMaster の
docs/LOCKED_STATE.md
を BowlingMaster200 に継承する。
OCRコア変更は、
LOCKED_STATE.md に定める解除手順を経た場合のみ許可する。
________________________________________
3-3. OCRは補助機能である
OCR結果を無条件に保存してはならない。
利用者が確認し、
必要に応じて修正できることを必須要件とする。

6. Adapter設計
   OCRコアとアプリ本体を直接接続しない。
   必ず Adapter を介して接続する。
________________________________________

・OcrSheetDraft.kt
・OcrToFrameMapper.kt
________________________________________
6-2. OcrSheetDraft
OCR結果を保持するDTO。
DB非依存。
保持内容
・activeFrame
・symbol
・outcome
・pins
・rawOcrText
________________________________________
6-3. OcrToFrameMapper
OCR結果を
BowlingMaster200入力モデルへ変換する。
責務
・OCR → FrameInputModel
・validation
・fallback
・変換エラー吸収
________________________________________
6-4. Adapter設計方針
OCRコアは変更しない。
必要な補正は
OcrToFrameMapper
で実施する。
OCR側へ逆流する修正は禁止する。

9. OCR失敗時フォールバック
   9-1. 基本方針
   OCRは補助機能である。
   OCR失敗時でも
   アプリは利用可能でなければならない。
________________________________________
9-2. 部分失敗
例
・記号のみ失敗
・残ピンのみ失敗
・一部セルのみ失敗
対応
・該当セルを空欄表示
・手修正可能
・保存強制禁止
________________________________________
9-3. 完全失敗
OCR結果が取得できない場合
即座に手入力画面へ移行する。
クラッシュは禁止する。

12-4. 入力記号
X = ストライク
/ = スペア
G = ガター
F = ファール
- = ミス
  0ボタンは実装しない。
________________________________________
12-5. 自動カーソル移動
入力後は次セルへ自動移動する。
ストライク入力時は
次フレーム1投目へ移動する。
________________________________________
12-6. DEL動作
現在セルに値がある場合
→ 現在セル削除
現在セルが空の場合
→ 前セルへ移動して削除
________________________________________
12-7. 入力バリデーション
不正入力は禁止する。
例
9 の後に 8
→ 禁止
7 の後に /
→ 許可
X の後に /
→ 禁止

12-9. 10フレーム入力ルール
10Fのみ3投目を持つ。
以下の場合のみ3投目を許可する。
・スペア成立
・ストライク成立
例

→ 終了

→ 可
X 8 1
→ 可
X X X
→ 可
________________________________________
12-10. OCR修正モード
OCR結果は入力済みデータとして表示する。
利用者は任意セルを修正可能。
修正後は自動的に次セルへ移動する。
DEL動作は通常入力と共通とする。
________________________________________
13. 禁止事項
    13-1. OCRコア変更禁止
    以下の変更を禁止する。
    ・OCRコア直接修正
    ・Layout比率変更
    ・OCR閾値変更
    ・activeFrame推定ロジック変更
    ・Symbol解析ロジック変更
    ・SpareOutcomeRules変更
    例外は
    LOCKED_STATE.md
    に定める解除手順を経た場合のみとする。
________________________________________
13-2. 強制確定禁止
OCR結果を利用者確認なしに保存してはならない。
OCRは補助機能であり、
最終判断は利用者が行う。
________________________________________
13-3. 不正データ保存禁止
以下のような不正データの保存を禁止する。
例
・1投目＋2投目が10を超える
・10Fルール違反
・未完成フレーム
・不正記号
保存前に必ず検証を行う。
________________________________________
13-4. クラッシュ禁止
OCR失敗
DBエラー
入力エラー
が発生しても、
アプリは異常終了してはならない。
可能な限り復旧またはフォールバックを行う。
